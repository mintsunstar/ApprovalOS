## 1. 한 줄 결론

원안(Gemini·이미지·아이템 단위·새 스키마)은 **방향(실 AI + 키 서버측)** 은 맞지만, **현재 ApprovalOS 구현과 스키마·UI·런타임이 맞지 않아 그대로 붙여넣기 진행하면 깨집니다.**  
권장: **현재 제품 스키마를 유지한 채 Gemini로 교체(경로 A)** 를 1차로 하고, 이미지 비전 분석은 2차로 분리한다.

## 2. 현재 개발 상황 (사실)

| 항목 | 현재 |
|------|------|
| 화면 | `/projects/:id/analysis` — 키워드 · 시안별 요약 · 감성 도넛 · 브랜드 적합도 바 |
| 실행 | WS admin만 · `mockAnalyzeProject`만 호출 (Edge 미호출) |
| 입력 | **텍스트**: 댓글·핀·투표 집계 (**이미지 미전송**) |
| 저장 | `ai_analyses` **프로젝트당 1건** (`localApi.saveAnalysis` upsert) |
| 스키마 | `keywords`, `item_summaries`, `overall_summary`, `sentiment`, `brand_fit_scores` |
| Edge 초안 | `supabase/functions/analyze-project` — **Claude** + 위와 **동일** 프로젝트 JSON |
| 프론트 헬퍼 | `analyzeProject()` → `analyze-project` invoke 존재하나 UI 미사용 |
| Supabase | 프론트 **미연결** (로컬 데모). `isSupabaseConfigured` 경로 있음 |
| AI 횟수 제한 | **없음** (Free 3 / Starter 10 가정은 거짓) |
| 플랜 | `free` / `pro` / `enterprise` — Starter 없음 |
| `brandGuideline` | 프로젝트 필드 **없음** |
| 이미지 URL | IndexedDB 참조 / data URL — **공개 HTTPS URL 아님** |

## 3. 원안 vs 현재 — 갭 분석

| # | 원안 가정 | 현재 실제 | 영향 |
|---|-----------|-----------|------|
| 1 | 분석 단위 = **시안 아이템 + 이미지** | 단위 = **프로젝트 + 텍스트** | 요청/응답·UI 전면 불일치 |
| 2 | 응답 = summary / strengths / improvements / brandConsistency | 응답 = keywords / sentiment / item_summaries / brand_fit_scores | **스키마·화면 전면 개편** 필요 (원안 §8 “UI·DB 건드리지 말 것”과 **모순**) |
| 3 | Edge명 `analyze-design-item` 신설 | 이미 `analyze-project`(Claude) 존재 | 신설 vs 교체 선택 필요 |
| 4 | `imageUrl` HTTPS fetch | 로컬 파일은 공개 URL 없음 | Storage 공개 URL 또는 base64 업로드 선행 필요 |
| 5 | AI 월 한도 프론트에 “이미 존재” | **미구현** | Edge 전에 체크할 로직 없음 |
| 6 | `brandGuideline` | 없음 | 필드 추가 또는 항상 null |
| 7 | Supabase Secrets + Edge 배포 | 로컬만 동작 | **사용자**: Supabase 프로젝트·키·배포 필수 |
| 8 | “mock 호출부만 교체” | mock 입출력이 원안과 다름 | 교체만으로는 원안 스펙 충족 불가 |
| 9 | Claude → Gemini | Edge는 Claude, 제품 문서도 Claude 언급 | 모델 전환 자체는 타당 (비용·비전) |

**원안에서 유지할 가치**

- 프론트 직접 Gemini 호출 금지 · Secrets에 키
- `responseMimeType` + `responseSchema` 강제
- 타임아웃·429·키 미설정 에러 처리
- `GEMINI_MODEL` 환경변수 분리
- `maxOutputTokens` 상한

## 4. 권장 전략

### 경로 A — 현재 제품에 맞춘 Gemini 전환 (**권장 · 1차**)

목표: **기존 UI·`ai_analyses`·알림을 유지**하고, mock/Claude를 Gemini로 바꾼다.

- Edge: 기존 `analyze-project`를 Gemini로 재작성 (또는 내부 프로바이더만 교체)
- 요청: 현재와 동일 — `projectId` + title/items/comments/pins/votes
- 응답: **기존 `AnalyzeProjectResult` JSON**을 `responseSchema`로 강제
- 프론트: `ProjectAnalysis`에서 `isSupabaseConfigured`면 `analyzeProject()`, 아니면 mock fallback
- 이미지 비전: **이번 라운드 제외**

→ 원안 §8(“UI·DB 변경 금지”)과 **유일하게 양립**하는 경로.

### 경로 B — 원안 그대로 (아이템 비전 분석 · **2차**)

목표: 시안 이미지를 Gemini에 넣어 강점/개선점 등 시각 분석.

선행 조건:

1. Storage(또는 서명 URL)로 **접근 가능한 imageUrl**
2. DB: 아이템별 분석 테이블 또는 JSON 확장 + **UI 전면 개편**
3. (선택) `brand_guideline` 컬럼
4. (선택) 플랜별 AI 횟수 — `plan_limits` 확장 + 강제

→ 별도 스프린트. 경로 A 완료 후 착수.

### 경로 C — 비권장

원안 프롬프트를 수정 없이 Cursor에 붙여넣고 진행 → UI/스키마 충돌·로컬에서 이미지 fetch 실패·한도 로직 부재로 스모크 실패 가능.

## 5. 진행계획 (경로 A 기준)

### Phase 0 — 전제 · 사용자 결정 (구현 전)

| # | 할 일 | 담당 |
|---|--------|------|
| D1 | 경로 **A(1차) / B(함께) / A만** 선택 | 사용자 |
| D2 | Supabase 프로젝트 생성 여부 (없으면 Edge 실호출 불가 → mock만) | 사용자 |

### Phase 1 — Edge Function (Gemini · 프로젝트 스키마)

| # | 작업 | 산출 |
|---|------|------|
| A1 | `analyze-project`를 Gemini 호출로 교체 (`GEMINI_API_KEY`, `GEMINI_MODEL`) | Edge 코드 |
| A2 | `responseSchema` = 기존 `AnalyzeProjectResult` 필드 | 구조화 JSON |
| A3 | 타임아웃 10s·429 재시도·키 미설정 500 · 파싱 실패 처리 | 원안 §5 이식 |
| A4 | `maxOutputTokens` ≤ 500~800 (프로젝트 JSON이 원안보다 큼 → **800 권장**) | 비용 |
| A5 | Claude 전용 코드·시크릿 문서에서 `GEMINI_*`로 갱신 | docs / 점검표 |

> 원안의 새 함수명 `analyze-design-item`은 **경로 B**에서만 사용. 경로 A는 기존 이름 유지가 프론트 `claude.ts` invoke와 맞음.

### Phase 2 — 프론트 연동 (최소 변경)

| # | 작업 | 산출 |
|---|------|------|
| F1 | `ProjectAnalysis.runAnalysis`: configured면 `analyzeProject()`, 아니면 mock | UI 동일 |
| F2 | `success:false` / throw → 기존 `toast.error` | 신규 UI 없음 |
| F3 | `claude.ts` 파일명·주석을 `aiAnalyze.ts` 등으로 정리(선택) | 가독성 |
| F4 | 저장은 계속 `localApi.saveAnalysis` **또는** Edge upsert + 프론트 get — **로컬 데모는 localApi 유지**, Edge upsert는 Supabase 전환 시 | 이중 저장 주의 |

**로컬 데모 주의:** Edge가 `ai_analyses`에 upsert해도 프론트는 `localDb`를 보므로, **데모 단계에서는 Edge가 JSON만 반환하고 저장은 프론트 `saveAnalysis`** 가 맞다. (현재 Claude Edge의 DB upsert는 프로덕션용 — 경로 A에서도 “응답만 반환, 저장은 클라이언트/추후 API”로 정리 권장.)
