## 2. 설계 (DB·플로우·역할)

### 2.1 멤버십 모델

| 개념 | 설계 | 근거 |
|------|------|------|
| 워크스페이스 멤버 | **`workspace_members` 테이블 없음** — `users.workspace_id`로 소속 표현 | `003_create_users.sql`, DB명세서 §3.2 |
| 프로젝트 멤버 | `project_members(project_id, user_id, role)` + `UNIQUE(project_id, user_id)` | `004_create_projects.sql`, DB명세서 §3.4 |
| 워크스페이스 링크 초대 | `workspaces.invite_token` (UNIQUE, random hex) | `002_create_workspaces.sql`, DB명세서 §3.1 |
| 이메일 초대 | `invitations` (workspace_id 필수, project_id 선택) | `014_create_invitations.sql`, DB명세서 §3.14 |

> `workspaces.invite_token`과 `invitations`는 **별개 경로**이다.  
> WS 링크 초대 = workspace 토큰, 이메일 초대 = invitations 테이블.

### 2.2 역할(Role)

```sql
-- 001_create_enums.sql
CREATE TYPE user_role AS ENUM ('admin', 'reviewer', 'approver', 'viewer');
```

| 적용 위치 | 컬럼 | 기본값 |
|-----------|------|--------|
| 워크스페이스 | `users.role` | `reviewer` |
| 프로젝트 | `project_members.role` | `reviewer` |

프론트 타입: `src/types/index.ts` — `UserRole`, `Invitation`, `ProjectMember`, `ROLE_LABELS`

### 2.3 초대·수락 플로우 (설계 의도)

```mermaid
flowchart LR
  A[관리자] --> B{초대 방식}
  B --> C[WS invite_token 링크]
  B --> D[invitations 이메일+token]
  C --> E[/invite/:token]
  D --> E
  E --> F[로그인/가입]
  F --> G[acceptInvitation]
  G --> H[users.workspace_id + role 설정]
  G --> I[project_members 추가]
```

### 2.4 RLS (설계만, 정책 불완전)

`016_rls_policies.sql` 기준:

| 테이블 | RLS | 정책 |
|--------|-----|------|
| users | ON | 동일 WS 멤버 SELECT, 본인 UPDATE |
| workspaces | ON | WS 멤버 SELECT, admin UPDATE |
| projects | ON | WS 단위 ALL + 공개 link SELECT |
| **project_members** | ON | **정책 없음** |
| **invitations** | ON | **정책 없음** |

> 정책이 없는 테이블은 RLS ON 시 접근이 막힐 수 있다. 프로덕션 적용 전 정책 보강 필요. (DB명세서 §5)

### 2.5 Edge Function (스텁)

`supabase/functions/send-invitation/index.ts` — Resend/SendGrid 연동 없이 `console.log`만 수행.

## 3. 데모(로컬) 구현 현황

> 런타임: `localStorage`(`approvalos_db_v1`) + IndexedDB(`approvalos_files`)  
> API 진입점: `src/lib/localDb.ts` → `localApi`

### 3.1 localApi — 멤버·초대 관련

| 기능 | 상태 | 설명 |
|------|------|------|
| `LocalDB` 스키마 | ✅ | `invitations`, `project_members` 포함 |
| `signup` | ✅ | 신규 WS + admin 사용자 생성 |
| `getWorkspaceMembers` | ✅ | `users.workspace_id` 필터 |
| `updateUser` | ✅ | 역할 변경, `workspace_id: null`로 제거 |
| `createInvitation` | ✅ | token 자동 생성 (16자), 호출측에서 7일 만료 설정 |
| `getInvitation` | ✅ | token으로 조회 + project/inviter join |
| `acceptInvitation` | ✅ | WS 소속·역할 설정, project_id 있으면 `project_members` 추가 |
| `createProject` | ✅ | 생성자를 `project_members` admin으로 등록 |
| `getProjectMembers` 등 CRUD | ❌ | 추가·제거·역할 변경 API 없음 |

### 3.2 프론트엔드 UI

| 화면 | 경로 | 구현 내용 |
|------|------|-----------|
| 워크스페이스 설정 | `WorkspaceSettings.tsx` | admin 전용, 멤버 목록·역할 변경·삭제, WS 초대 링크 복사 |
| 초대 수락 | `InviteAccept.tsx` | `/invite/:token`, `invitations` token 수락 UI |
| 프로젝트 생성 시 초대 | `ProjectNew.tsx` | 이메일+역할 입력 → `createInvitation` |
| 프로젝트 설정 | `ProjectSettings.tsx` | **멤버/초대 섹션 없음** (메타·공개 링크만) |
| 라우트 | `App.tsx` | `/invite/:token` 등록 |
| 네비 | `AppLayout.tsx` | "멤버 · 설정" → `/workspace/settings` |

### 3.3 역할 기반 UI (워크스페이스 `users.role` 기준)

| 화면/기능 | 접근 조건 |
|-----------|-----------|
| `WorkspaceSettings` | admin |
| `ProjectSettings` | admin 또는 `created_by` |
| `ApprovalCenter` | approver 또는 admin |
| `ProjectMain`, `ProjectVote` 등 | admin/`created_by` 분기 |

### 3.4 표시용 멤버 카운트

- `enrichProject()`에서 `project_members.length` → `member_count`
- `ProjectMain`, `ProjectCard`, `ProjectReport`에서 "참여자 N명" 표시

## 4. 누락·미완성 (갭 목록)

### 4.1 초대 플로우 (치명적)

| # | 갭 | 설명 | 근거 |
|---|-----|------|------|
| 1 | **WS 링크 초대 수락 불가** | `WorkspaceSettings`는 `/invite/{workspace.invite_token}` 복사하지만, `InviteAccept`는 `invitations.token`만 조회. fallback 주석만 있고 에러 처리 | `InviteAccept.tsx` 43–46행 |
| 2 | **`?invite=` 가입 미연동** | `InviteAccept` → `/signup?invite=…` 링크 있으나 Signup/Login이 query param 미처리, 가입 후 `acceptInvitation` 미호출 | `Login.tsx`, 작업점검표 §2-A #10 |
| 3 | **이메일 발송 없음** | `ProjectNew`에서 invitation 레코드만 생성. URL 표시·Edge 호출·메일 없음 | `ProjectNew.tsx` 92–105행 |
| 4 | **초대 링크 UI 없음** | 프로젝트 생성 후 초대받은 사람이 쓸 `/invite/{token}` 링크를 관리자에게 보여주지 않음 | `ProjectNew.tsx` InviteSection |

### 4.2 멤버 관리

| # | 갭 | 설명 |
|---|-----|------|
| 5 | **프로젝트 멤버 관리 UI 없음** | `ProjectSettings`에 멤버 섹션 없음 |
| 6 | **project_members CRUD API 없음** | `getProjectMembers`, `addProjectMember`, `removeProjectMember`, `updateProjectMemberRole` 등 없음 |
| 7 | **접근 제어 미적용** | 대시보드는 WS 전체 프로젝트 표시. `project_members`로 필터링·권한 검사 없음 |

## 5. 설계 vs 프론트 불일치

| DB 설계 | 프론트 현실 |
|---------|-------------|
| WS `invite_token` 링크 초대 | UI는 링크 생성, **수락 로직 미구현** |
| `invitations.email` | 저장만, **수락 시 검증 없음** |
| `project_members.role` | **표시·접근제어에 미사용** (카운트만) |
| `users.role` (WS) | **대부분의 권한 분기에 사용** |
| `workspace_members` (없음) | `getWorkspaceMembers`로 `users` 필터 — **일치** |

## 6. 플로우 다이어그램 (설계 vs 현실)

```mermaid
flowchart TB
  subgraph designed [설계된 플로우]
    A1[관리자] --> B1{초대 방식}
    B1 --> C1[WS invite_token]
    B1 --> D1[invitations 이메일]
    C1 --> E1[/invite/:token]
    D1 --> E1
    E1 --> F1[로그인/가입]
    F1 --> G1[acceptInvitation]
    G1 --> H1[WS 소속 + project_members]
  end

  subgraph actual [현재 구현]
    C2[WS 링크 복사] -.->|수락 로직 없음| X1[에러]
    D2[invitation 레코드만] -.->|링크/메일 없음| X2[사용자 모름]
    F2[Signup ?invite=] -.->|미처리| X3[초대 무시]
  end
```
